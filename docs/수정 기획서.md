# 나랑 싸울래? — 개발 기획서

---

## 1. 서비스 구조 요약

AI 판사가 커플/인간관계 갈등을 재판 형식으로 판결해주는 **재미형 콘텐츠 앱**.

**수익 모델: 완전 무료 + 보상형 광고(Rewarded Ad)**

사용자는 광고를 시청하여 **토큰(Token)** 을 획득하고, 토큰을 소비하여 기능을 사용한다.
과금 없음. 구독 없음. 프리미엄 플랜 없음.

---

## 2. 토큰 시스템

### 토큰이란?

광고 시청으로 얻는 인앱 재화. 현금 구매 불가.

> 광고 시청 구현은 **토스 앱인토스 SDK**를 사용한다.
> SDK 연동 방법 및 보상형 광고 콜백 처리는 앱인토스 개발자 문서를 참고할 것.
> 서버사이드 검증(광고 완료 콜백)을 반드시 사용하여 어뷰징을 방지한다.

### 토큰 획득

| 방법                    | 획득량  |
| ----------------------- | ------- |
| 보상형 광고 시청 (30초) | +3 토큰 |
| 앱 첫 실행 (1회)        | +5 토큰 |
| 일일 출석               | +1 토큰 |

### 토큰 소비

| 기능                  | 소비량 |
| --------------------- | ------ |
| 판결 결과 확인        | 2 토큰 |
| 판결문 상세 보기      | 1 토큰 |
| AI 변호사 선임        | 2 토큰 |
| 스스로 변호 결과 확인 | 1 토큰 |
| 항소 (1회)            | 2 토큰 |
| 프리미엄 판사 사용    | 3 토큰 |

> **설계 원칙:** 광고 1회(3토큰) = 판결 1회(2토큰) + 여유 1토큰.
> 사용자가 광고를 너무 많이 봐야 한다는 느낌이 들지 않도록 토큰 가성비를 유지한다.

## 3. 사용자 플로우

```
앱 진입
  └─ 내 입장 입력 (최대 100자)
  └─ 상대 입장 입력 (최대 100자)
  └─ 판사 선택
        └─ 기본 판사: 무료
        └─ 프리미엄 판사: 3토큰 차감
  └─ 판결 생성 중 (AI 호출)
  └─ 결과 확인 → 2토큰 차감
        ├─ 토큰 충분: 바로 확인
        └─ 토큰 부족: 광고 시청 → 토큰 획득 → 확인
  └─ 판결문 보기 → 1토큰
  └─ 변호사 선임 → 2토큰
  └─ 항소 → 2토큰 (1회 한정)
  └─ 공유 / 저장 (무료)
```

## 4. 핵심 기능 정의

### 4-1. 판결 생성

**입력값**

- `user_claim`: 내 입장 (최대 100자)
- `opponent_claim`: 상대 입장 (최대 100자)
- `judge_id`: 선택한 판사

**출력값**

- `user_fault`: 내 과실 % (0~100)
- `opponent_fault`: 상대 과실 %
- `comment`: 판사 코멘트 (최대 40자)
- `verdict_detail`: 판결문 전문 (판결문 보기 시 노출)

**판결 생성은 토큰 차감 전에 백그라운드에서 미리 실행**한다.
결과는 생성 완료 후 DB에 저장되며, 토큰 차감 시점에 노출된다.
(사용자가 광고를 보고 있는 동안 이미 판결이 완성된 상태여야 UX가 자연스럽다.)

---

### 4-2. 판사 시스템

판사는 AI 프롬프트 페르소나다. 같은 사건도 판사마다 다른 결과가 나온다.

#### 판사 구분

판사는 두 종류로 구분되며, UI에서 반드시 분리하여 표시한다.

| 구분          | 설명                        | DB 필드                |
| ------------- | --------------------------- | ---------------------- |
| **공식 판사** | 서비스에서 직접 등록한 판사 | `isUserCreated: false` |
| **유저 판사** | 일반 사용자가 등록한 판사   | `isUserCreated: true`  |

**UI 분리 방식**

- 판사 선택 화면에서 탭 또는 섹션으로 "공식 판사" / "유저 판사" 구분
- 유저 판사는 등록자 닉네임 표시
- 판사 카드에 뱃지로 구분 (`공식` / `유저`)
- 각 탭 내에서는 추천수 기준 내림차순 정렬

---

**공식 판사 (무료)**
| 이름 | 스타일 |
|------|--------|
| 기본 판사 | 중립적, 논리 중심 |
| 공감형 판사 | 감정에 공감, 피해자 중심 |
| 냉정한 판사 | 팩트만, 감정 없음 |
| 초강력 T 판사 | 극단적 논리, 공감 제로 |

**프리미엄 판사 (3토큰)**

- 특이한 말투, 강한 캐릭터
- 밈 요소 강화 (예: 조선시대 판관, 외계인 판사 등)
- `isPremium: true`로 구분

---

#### 유저 판사 등록

사용자는 직접 판사를 만들어 서비스에 등록할 수 있다.

**등록 프로세스**

MBTI 검사처럼 단계별로 항목을 선택/입력하는 방식으로 판사를 구성한다.
입력이 완료되면 AI가 자동으로 시스템 프롬프트를 생성한다.

**단계별 질문 구성**

```
Step 1. 판사 이름
  → 직접 입력 (최대 10자)
  예시: 황판사, 감성 충만 이판사

Step 2. 판단 기준
  → 선택 (1개)
  A. 논리와 팩트 중심
  B. 감정과 공감 중심
  C. 상황에 따라 유동적

Step 3. 판결 성향
  → 선택 (1개)
  A. 원고(나) 편
  B. 피고(상대) 편
  C. 완전 중립
  D. 상황에 따라 다름

Step 4. 말투
  → 선택 (1개)
  A. 딱딱하고 공식적인 말투
  B. 친근하고 부드러운 말투
  C. 독설적이고 직설적인 말투
  D. 재밌고 유머러스한 말투
  E. 직접 입력 (최대 30자)
     예시: "~이로다체", "반말 친구처럼"

Step 5. 판결 특징 (선택, 최대 2개)
  A. 피해자 입장을 우선 고려한다
  B. 감정보다 행동 결과를 따진다
  C. 변명의 여지를 거의 안 준다
  D. 양쪽 모두에게 책임이 있다고 본다
  E. 상황 맥락을 중요하게 생각한다

Step 6. 판사 한 줄 소개
  → 직접 입력 (최대 30자)
  예시: "공감은 없다, 팩트만 있다"
```

입력 완료 → AI가 시스템 프롬프트 자동 생성 → 미리보기 제공 → 등록 신청

**관리자 승인 프로세스 (AI 자동 검토)**

등록 신청 후 사람이 직접 검토하는 것이 아니라 **스케줄러가 주기적으로 AI에게 검토 요청**을 보내고 자동으로 승인/반려 처리한다.

- 승인 시: `isApproved: true`로 변경 → 전체 공개
- 반려 시: `isApproved: false` 유지, `rejectReason` 저장 → 사용자에게 사유 노출
- **승인 전까지 본인 포함 누구도 사용 불가**

**스케줄러 주기:** 10~30분 간격으로 `isApproved: false` && `rejectReason: null` 인 판사 일괄 처리

**AI 검토 기준**

스케줄러는 아래 기준을 시스템 프롬프트에 포함하여 AI에게 판사 이름, 한 줄 소개, 생성된 프롬프트를 전달한다.

```
[승인 기준]
다음 조건을 모두 충족하면 승인(approved)한다:
- 판사 이름이 욕설, 비하, 혐오 표현을 포함하지 않는다
- 특정 실존 인물(연예인, 공인, 일반인)을 직접 지칭하거나 비하하지 않는다
- 성적으로 부적절한 표현이 없다
- 서비스 취지(연애/인간관계 갈등 판결)에 부합하는 내용이다
- 광고성 문구나 외부 링크, 개인정보가 포함되지 않는다

[반려 기준]
하나라도 해당하면 반려(rejected)한다:
- 욕설, 비속어, 혐오 발언
- 특정 성별/인종/종교/지역에 대한 차별적 표현
- 실존 인물 실명 언급 및 비하
- 성적 표현
- 광고, 스팸, 개인정보

반환 형식 (JSON만 반환):
{
  "result": "approved" | "rejected",
  "reason": "반려 사유 (approved일 경우 null)"
}
```

#### 판사 등록 주의사항 (경고 문구)

유저 판사 등록 폼 하단에 다음 경고 문구를 항상 표시한다.
제재 시스템은 구현하지 않으며, 경고 문구로 대체한다.

> ⚠️ **등록 전 반드시 확인하세요**
> 욕설, 특정 인물 비하, 혐오 표현, 성적 내용이 포함된 판사는 자동으로 반려됩니다.
> 부적절한 내용 반복 등록 시 서비스 이용이 제한될 수 있습니다.

---

#### 추천/비추천 및 랭킹

- 판사 1인당 1회 추천 또는 비추천 가능 (중복 불가, 변경 가능)
- 랭킹 점수 = `upvotes - downvotes`
- 공식 판사와 유저 판사 각각 별도 랭킹 적용
- 사용 횟수(`usageCount`)도 랭킹 보조 지표로 활용 가능

---

### 4-3. 항소 시스템

- 판결 1건당 항소 1회 가능
- 항소 시 동일 판사 또는 다른 판사 선택 가능
- 항소 결과가 최종 판결
- 항소 후 추가 항소 불가

**DB에 `stage` 컬럼으로 관리**

- `initial`: 1차 판결
- `appeal`: 항소 (최종)

---

### 4-4. 변호 시스템

**AI 변호사 선임 (2토큰)**

- 내 입장을 강화하는 논리 3가지 생성
- 상대 논리 반박 포인트 생성
- 재판용 변론 문장 생성

**스스로 변호 (직접 작성)**

- 사용자가 직접 변론 입력
- 변론 포함 재판결 요청 → 결과 확인 시 1토큰

---

### 4-5. 공유 기능

공유는 서비스의 핵심 바이럴 수단이다. **토스 앱인토스 공유 SDK**를 최대한 활용하며, 판결별 동적 OG 이미지는 생성하지 않는다.

#### 공유 방식

**토스 공유 SDK 활용 (메인)**

- 앱인토스 SDK의 공유 기능을 사용하여 카카오톡, 문자 등 공유
- 공유 시 서비스 내 해당 판결 결과 페이지 딥링크 전달
- SDK 연동 방법은 앱인토스 개발자 문서를 참고할 것

**공유 URL 구조**

```
https://서비스도메인/result/{fightId}
```

공유 링크를 받은 사람이 접속하면 해당 판결 결과 페이지가 표시된다.
비로그인 상태에서도 결과 조회는 가능하다 (읽기 전용).

**공유 이미지 정책**

- 판결별 동적 이미지 생성 없음 (서버 부하 방지)
- 서비스 고정 대표 이미지(OG 메타 태그)를 사용
- 결과 페이지 URL 공유 시 고정 썸네일 + 판결 텍스트 미리보기로 충분

#### 공유 진입점

공유 기능은 서비스 전반에서 쉽게 접근할 수 있어야 한다.

| 진입점                | 공유 내용                                       |
| --------------------- | ----------------------------------------------- |
| 판결 결과 화면        | 이번 판결 결과 링크                             |
| 싸움 히스토리 각 항목 | 개별 판결 링크                                  |
| 판사 상세 화면        | 해당 판사 소개 링크 ("이 판사로 재판받기" 유도) |
| 내가 만든 판사        | 내 판사 공유 ("이 판사 써봐" 유도)              |

> 공유는 항상 무료. 토큰 소비 없음.
> 공유 링크로 유입된 신규 사용자가 바로 재판을 시작할 수 있도록 랜딩 흐름을 자연스럽게 연결한다.

## 5. 데이터 모델

```prisma
model User {
  id         String      @id @default(cuid())
  nickname   String
  token      Int         @default(5)
  createdAt  DateTime    @default(now())

  fights     Fight[]
  judges     Judge[]
  votes      JudgeVote[]
}

model Fight {
  id              String      @id @default(cuid())
  userId          String
  user            User        @relation(fields: [userId], references: [id])

  userClaim       String      @db.VarChar(100)
  opponentClaim   String      @db.VarChar(100)

  userFault       Int
  opponentFault   Int
  comment         String      @db.VarChar(40)
  verdictDetail   String?     @db.Text

  judgeId         String
  judge           Judge       @relation(fields: [judgeId], references: [id])

  stage           FightStage  @default(INITIAL)
  defense         String?     @db.Text

  createdAt       DateTime    @default(now())
}

enum FightStage {
  INITIAL
  APPEAL
}

model Judge {
  id            String      @id @default(cuid())
  name          String
  description   String      @db.VarChar(30)
  prompt        String      @db.Text

  isUserCreated Boolean     @default(false)
  isPremium     Boolean     @default(false)
  isApproved    Boolean     @default(false)
  rejectReason  String?     // AI 반려 시 사유 저장
  createdBy     String?

  upvotes       Int         @default(0)
  downvotes     Int         @default(0)
  score         Int         @default(0)
  usageCount    Int         @default(0)

  createdAt     DateTime    @default(now())

  fights        Fight[]
  votes         JudgeVote[]

  @@index([score])
  @@index([isUserCreated, isApproved, score])
}

model JudgeVote {
  id        String    @id @default(cuid())
  judgeId   String
  judge     Judge     @relation(fields: [judgeId], references: [id])
  userId    String
  user      User      @relation(fields: [userId], references: [id])
  isUpvote  Boolean
  createdAt DateTime  @default(now())

  @@unique([judgeId, userId])
}

model TokenLog {
  id        String      @id @default(cuid())
  userId    String
  amount    Int
  reason    TokenReason
  createdAt DateTime    @default(now())
}

enum TokenReason {
  SIGNUP_BONUS
  DAILY_LOGIN
  AD_REWARD
  VERDICT_VIEW
  VERDICT_DETAIL
  LAWYER_AI
  DEFENSE_RESULT
  APPEAL
  PREMIUM_JUDGE
}
```

## 6. API 설계

### 판결 관련

| Method | Path                      | 설명                   |
| ------ | ------------------------- | ---------------------- |
| POST   | `/api/fights`             | 판결 생성 요청         |
| GET    | `/api/fights/:id`         | 판결 조회              |
| POST   | `/api/fights/:id/reveal`  | 토큰 차감 후 결과 공개 |
| POST   | `/api/fights/:id/appeal`  | 항소 요청              |
| POST   | `/api/fights/:id/defense` | 변호 요청              |

### 판사 관련

| Method | Path                        | 설명                                        |
| ------ | --------------------------- | ------------------------------------------- |
| GET    | `/api/judges`               | 판사 목록 전체 (공식/유저 구분, 랭킹순)     |
| GET    | `/api/judges?type=official` | 공식 판사 목록                              |
| GET    | `/api/judges?type=user`     | 유저 판사 목록                              |
| POST   | `/api/judges`               | 유저 판사 등록 신청                         |
| GET    | `/api/judges/:id`           | 판사 상세                                   |
| POST   | `/api/judges/:id/vote`      | 추천/비추천 (body: `{ isUpvote: boolean }`) |
| DELETE | `/api/judges/:id/vote`      | 추천/비추천 취소                            |
| GET    | `/api/judges/my`            | 내가 등록한 판사 목록                       |

### 토큰 관련

## 11. 주의사항

**광고 SDK**
광고 시청 및 토큰 지급은 **토스 앱인토스 SDK**를 통해 구현한다.
SDK 문서의 보상형 광고 서버사이드 콜백 방식을 사용하여 클라이언트 단 어뷰징을 방지한다.
`/api/tokens/ad-reward`는 SDK 콜백에서만 호출되도록 설계한다.

**공유 SDK**
토스 앱인토스 공유 SDK 사용법은 앱인토스 개발자 문서를 참고한다.
공유 링크(`/result/:id`)는 비로그인 상태에서도 읽기 전용으로 열람 가능해야 한다.
공유 링크 유입 사용자가 재판 시작 버튼을 쉽게 찾을 수 있도록 CTA를 결과 페이지 하단에 배치한다.

**판사 AI 검토 스케줄러**
`/api/scheduler/judge-review`는 외부에서 직접 호출되지 않도록 `Authorization` 헤더로 보호한다.
스케줄러는 홈 서버 크론탭 또는 외부 스케줄러 서비스(Vercel Cron, GitHub Actions 등)로 구성한다.
한 번에 처리할 건수에 상한선을 두어 AI API 비용이 몰리지 않도록 한다. (예: 1회 최대 10건)

**판사 랭킹 score 동기화**
`upvotes`, `downvotes` 변경 시 `score = upvotes - downvotes`도 함께 업데이트한다.

**AI 응답 안정성**
JSON 파싱 실패 시 재시도 로직 필요. `userFault + opponentFault = 100` 검증 필수.

**과실 % 합산 정규화**

```ts
const total = userFault + opponentFault;
const normalizedUserFault = Math.round((userFault / total) * 100);
const normalizedOpponentFault = 100 - normalizedUserFault;
```
